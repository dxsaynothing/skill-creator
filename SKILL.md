# 测试用例生成器 (Test Case Generator)

根据用户提供的需求文档和接口文档，自动分析并生成结构化的测试用例，输出为可在 XMind 中打开的 `.xmind` 格式文件。

## 工作流程

### 1. 收集输入文档

**你需要主动向用户明确：**

1. **需求文档** — 功能描述、业务规则、用户故事、验收标准等
2. **接口文档** — API 接口定义、请求/响应格式、字段说明、状态码等

如果用户没有提供完整的文档，先引导用户提供，或询问具体需求进行拆解分析。

### 2. 分析文档

读取并分析用户提供的文档，识别：

- **功能模块** — 系统有哪些核心功能模块
- **业务规则** — 每个功能的关键业务流程和规则
- **接口定义** — 每个接口的方法、路径、参数、响应
- **数据约束** — 字段类型、长度限制、必填项、枚举值
- **状态流转** — 业务状态的合法/非法转换路径
- **权限控制** — 不同角色/权限下的接口访问差异
- **异常场景** — 可能的错误输入、超时、并发、依赖服务异常等

### 3. 设计测试用例

基于分析结果，为每个功能模块/接口设计测试用例，覆盖以下维度：

| 维度 | 说明 |
|------|------|
| **正常流程** | 最典型的成功路径，核心业务流程 |
| **边界值** | 参数的最小值、最大值、临界值 |
| **异常/负面** | 无效输入、非法操作、权限不足 |
| **组合场景** | 多个功能联动的业务流程 |
| **数据完整性** | 创建/查询/更新/删除各环节数据一致 |
| **幂等性** | 重复请求的处理结果一致 |
| **并发** | 同时操作、资源竞争 |

### 4. 生成 XMind 文件

使用 `scripts/generate_xmind.py` 脚本生成 `.xmind` 文件。

#### 运行方式

```bash
python skill-creator/scripts/generate_xmind.py \
  --output /path/to/output.xmind \
  --title "项目名称-测试用例"
```

#### 输入数据格式

在调用脚本前，你需要先将设计的测试用例整理为 JSON 格式，通过标准输入传递给脚本：

```python
# 脚本从 stdin 读取 JSON，结构如下：
{
  "title": "项目名称-测试用例",
  "modules": [
    {
      "name": "模块名称（如：用户管理）",
      "groups": [
        {
          "name": "测试分组（如：正常功能测试）",
          "cases": [
            {
              "id": "TC-001",
              "title": "测试用例标题",
              "precondition": "前置条件（可选）",
              "steps": ["步骤1", "步骤2"],
              "expected": "预期结果",
              "priority": "P0/P1/P2/P3",
              "type": "功能测试/接口测试/边界值/异常场景/集成测试"
            }
          ]
        }
      ]
    }
  ]
}
```

**优先级定义：**
- **P0** — 核心功能，冒烟测试必过
- **P1** — 重要功能，业务主流程
- **P2** — 次要功能，异常场景
- **P3** — 边界值、易用性、边缘场景

## 输出结构

生成的 XMind 文件结构如下：

```
项目名称-测试用例
├── 模块A
│   ├── 正常功能测试
│   │   ├── TC-001 测试用例标题
│   │   │   ├── ⚡前置条件: xxx
│   │   │   ├── 📋步骤: 1. xxx 2. xxx
│   │   │   ├── ✅预期结果: xxx
│   │   │   ├── 🏷优先级: P0
│   │   │   └── 🏷类型: 功能测试
│   │   ├── TC-002 ...
│   ├── 边界值测试
│   ├── 异常场景测试
├── 模块B
│   ├── ...
```

## 用例设计原则

1. **完整性** — 覆盖正常、异常、边界场景，不留盲区
2. **独立性** — 每条用例独立可执行，不依赖其他用例结果
3. **可追溯** — 每条用例可追溯到对应的需求或接口
4. **清晰性** — 标题简洁明了，步骤和预期结果无歧义
5. **优先级明确** — 按重要程度标注 P0-P3

## 注意事项

- 文档读取时关注业务逻辑和数据流，不要遗漏约束条件
- 接口测试用例要覆盖不同的参数组合（必填+可选、有效+无效）
- 对于状态机类的业务（如订单状态流转），画出状态转换图作为用例基础
- 如果用户提供了已有的用例模板或特殊要求，按用户要求调整输出格式
- 脚本生成成功后，告知用户文件路径，并建议用 XMind 打开查看
